================================================================================
  1. VISION & GOALS
================================================================================

Nordvest is a statically typed, compiled language that combines readable
indentation-based syntax with first-class mathematical notation, strong null
safety, expressive pattern matching, and practical systems-programming features
(inline assembly, GPU kernels, C/C++ interop).

Core design axioms:
  - Readability first: syntax should read like well-written pseudocode.
  - Zero-cost abstractions: high-level constructs compile to efficient native
    code without hidden overhead.
  - Correctness by default: null safety, exhaustive match, Result<T> propagation,
    and checked I/O errors eliminate whole classes of runtime bugs.
  - Unified tooling: a single "nv" binary handles compile, run, test, fmt, doc,
    and package management — no Makefiles, no build scripts for common cases.
  - Mathematical expressiveness: ∀ ∃ ∑ ∏ ∈ ∧ ∨ ¬ π e ∞ are first-class syntax,
    not library calls.
  - Progressive escape hatches: pure Nordvest by default, inline asm and C/C++
    interop when needed, GPU kernels via @gpu annotation.

Non-goals (v1):
  - Dynamic/runtime typing or reflection-heavy metaprogramming.
  - Garbage collection as the primary memory strategy (ownership/RC TBD post-v1).
  - Competing with C in raw systems-level kernel programming.


================================================================================
  2. PHASED IMPLEMENTATION ROADMAP
================================================================================

The roadmap follows six major phases. Each phase ends with a named milestone
that is releasable (even if incomplete). The bootstrap compiler is written in
Kotlin and is retained permanently; Phase 5 delivers a real standard library
and Phase 6 delivers IDE plugins.

  PHASE 0 — FOUNDATION
  Goal: project skeleton, tooling scaffolding, decision record

Milestone: M0 "Scaffolding"

────────────────────────────────────────────────────────────────────────────────
  PHASE 1 — BOOTSTRAP CORE
  Goal: compile and run the full syntax of EXAMPLE.txt sections 1–27
  Language: Kotlin; target: LLVM IR → native binary
────────────────────────────────────────────────────────────────────────────────

Milestone: M1 "Hello World to Modules"

  Language features implemented in this phase:
    - Lexer/scanner for all tokens including Unicode operators (∀ ∃ ∑ ∏ → ∈ ∧
      ∨ ¬ π e ∞ ≤ ≥ ≠ ^ ÷ |> & | ~ ⊕ << >>).
      // (two slashes) is always a comment — there is no // integer-division token.
      ÷ (U+00F7) is the sole integer-division operator.
      xor is a keyword alias for ⊕; | is a single-token distinct from |> (maximal
      munch).
    - Indentation-sensitive parser (Python-style INDENT/DEDENT token injection).
    - Core types: int, int64, float, float32, float64, bool, str, byte, char.
    - Type aliases (type Meters = float).
    - let and var bindings with type inference (Hindley-Milner extended for OOP).
    - String interpolation: "step {count}" and formatted "{val:.2f}".
    - Raw strings: r"..." (no escapes, no interpolation); r"""...""" multi-line
      with indentation stripped to the level of the closing """.
    - Arithmetic, comparison, logical, and bitwise operators including Unicode
      variants. Compound assignments &= |= ⊕= <<= >>=.
    - Power operator ^.
    - Functions: fn, default parameters, named arguments, variadic (int...),
      first-class fn types, lambda syntax.
      Lambda forms: single-param (x → expr), multi-param ((x, y) → expr),
      zero-param (() → expr). Parameter types inferred from context; explicit
      annotation always allowed.
      Void function type: fn() → ().
      Trailing lambda: if the last argument is a lambda it may be written
      outside the call parens as { params → expr } or a multi-line { } block;
      parens omitted entirely when the lambda is the only argument.
      No currying or partial application — |> with _ positional hole covers
      functional composition without the type-inference cost of currying.
    - Inline if expression: if cond then a else b — usable anywhere an expression
      is expected. `then` keyword disambiguates from the if-statement form.
      No ternary operator (?:) — eliminated to remove ? disambiguation complexity.
    - Closure capture semantics: by-reference (RC) by default; explicit capture
      list [name] snapshots a value at closure-creation time; [copy name] for
      go-spawned coroutines. Compiler warns on implicit mutable capture in go.
    - Tuples: (T, T) return types, named tuple fields, destructuring at call site.
    - The return arrow → (interchangeable with ->).
    - Pipeline operator |> with _ positional hole.
    - Classes: primary constructor, body fields, init block, super.init.
    - Three-tier visibility: pub (exported), pub(pkg) (package-internal),
      (default) file-private.
    - Computed properties: fn name → T (no parameter list — property, not method);
      optional get/set blocks for settable properties.
    - Subscript operators: fn [](params) → T (getter) and fn []=(params, T) (setter).
    - Interfaces with default implementations.
    - Associated types in interfaces: type Name declaration; where C.Item: Constraint.
    - Inheritance (: BaseClass).
    - Enums: simple, raw-value, and sealed (discriminated unions with associated
      data). Sealed classes may be recursive (variants reference the enclosing
      sealed type), enabling linked lists, trees, and expression ASTs.
    - Structs: value types (struct keyword); stack-allocated, copy semantics, no
      inheritance. Complement to class (reference type) and record (immutable ref).
    - Records: immutable reference type with auto ==, str(), hash().
    - Null safety: T?, ?., ??, !., if let, guard let, chained if let.
      weak and unowned reference modifiers for breaking reference cycles.
    - Type tests and casts: is (boolean, narrows type via smart cast in then-branch),
      as? (explicit safe downcast, returns T?), as! (forced downcast, panics).
    - Arrays [T]: literals, indexing, slices with range notation.
    - 2D matrices [[T]]: A[row, col] indexing, row/column slice M[1,*].
    - Range notation [a,b], ]a,b[, [a,b[, ]a,b] in for loops, slices, and match.
    - List comprehensions: [expr for x in range if pred].
    - Quantifiers ∀ ∃ ∑ ∏ with : body syntax and → block syntax.
    - Built-in constants π, e, ∞.
    - Mathematical logical operators ∧ ∨ ¬ (with && || ! aliases).
    - for loops (range and collection iteration), while loops.
    - Labeled loops: @label for/while; break @label; continue @label.
    - defer statement: LIFO cleanup on scope exit, regardless of return path.
    - Pattern matching (match): literals, ranges, type patterns (positional and
      named-field destructuring), guard clauses, tuple patterns, wildcard _,
      or-patterns (A(_) | B(_) in one arm), nested patterns (Ok(Some(x))),
      exhaustiveness checking.
    - Result<T> / Result<T,E>: Ok, Err, ? propagation, getOrElse, getOrThrow.
      Result<T> defaults to Result<T, Error>; Error is a built-in interface with
      message() → str and cause() → Error? methods.
    - Checked exceptions as sugar: throws E desugars to a Result<T, E> return type;
      the two forms interoperate (a throws fn can be called with ? propagation).
    - try/catch/finally, throw for the throws form.
    - Extension functions (extend T) and retroactive conformance (extend T : Interface):
      declare that an existing type satisfies an interface without modifying it.
      Conformance is nominal and module-scoped; conflicting conformances for the
      same type+interface pair are a compile error.
    - Generics: generic functions and classes, single constraint, where clauses,
      associated types.
    - Modules: module declaration, pub/pub(pkg) visibility, import, import as alias.
      Multi-file modules: all .nv files in a directory with the same module
      declaration share a module; subdirectories form sub-modules. A mod.nv file
      controls which sub-modules are re-exported with pub import.
    - Conditional compilation: @if(pred) / @else at declaration scope; predicates
      platform, arch, debug, release, feature("name").
    - Doc comments /** ... */ (parsed and stored in AST for nv doc).
    - Comments: // and /* ... */.

  Compiler internals:
    1.1  Lexer
         - Hand-written recursive-descent or ANTLR4 grammar.
         - INDENT/DEDENT injection based on column offsets.
         - Unicode character categories for identifier start/continuation.
         - All Unicode operator tokens mapped to internal token kinds.

    1.2  Parser → AST
         - Recursive-descent parser following the PEG grammar.
         - Produces an immutable AST with source-location spans on every node.
         - Error recovery: insert synthetic nodes and continue to catch multiple
           errors in one pass.

    1.3  Name resolution & scope analysis
         - Module-level symbol table; scoped symbol tables for blocks.
         - Forward references within a module are allowed.
         - Resolve imports and build the module dependency graph.

    1.4  Type checker
         - Bidirectional type inference (synthesize + check modes).
         - Null safety flow analysis (nullable narrowing in if let / guard let).
         - Exhaustiveness checker for match (sealed, enum, bool, nullable).
           Must handle or-patterns and recursive sealed types (e.g. Tree<T>).
         - Generic instantiation and constraint satisfaction.
         - Tuple type inference including named fields.

    1.5  IR lowering (HIR → MIR → LLVM IR)
         - HIR (high-level IR): desugared AST; quantifiers expanded to loops,
           list comprehensions to explicit loop + accumulator, string
           interpolation to concat calls.
         - MIR (mid-level IR): SSA form; explicit move semantics placeholder
           (all values copied for now; ownership pass added later).
         - LLVM IR emission via llvm4j or LLVMSharp (JVM-accessible LLVM
           bindings). Alternatively, emit textual LLVM IR and invoke llc.

    1.6  Runtime support library (written in C, linked into every nv binary)
         - String representation (length-prefixed UTF-8 with small-string opt).
         - Array/slice representation (ptr + length + capacity).
         - Reference-counted heap allocator.
         - Panic handler (for ! forced-unwrap and assertion failures).
         - Result<T> boxing for mixed Ok/Err value representation.

    1.7  nv CLI — Phase 1 commands
         - nv run <file>    compile to temp binary and exec.
         - nv build <file>  compile to named output binary.
         - nv fmt <file>    format using canonical style (stub: just re-emit).

  Phase 1 test targets:
    - All 27 language-feature sections (1–27) of EXAMPLE.txt produce correct output.
    - At least 300 unit tests covering lexer, parser, type checker, and codegen.
    - "Hello World" benchmark: < 100 ms end-to-end compile time.

────────────────────────────────────────────────────────────────────────────────
  PHASE 2 — SYSTEMS & CONCURRENCY
  Goal: async/await, channels, inline asm, GPU, C/C++ interop, full stdlib v1
────────────────────────────────────────────────────────────────────────────────

Milestone: M2 "Production-Ready v0.1"

  Language features added:
    2.1  Concurrency model
         - Async runtime: M:N green-thread scheduler (libuv or custom epoll/kqueue
           event loop). One OS thread per CPU core; work-stealing scheduler.
           Every goroutine is a lightweight coroutine; channel/IO operations
           suspend the coroutine, not the OS thread.
         - go statement: fire-and-forget goroutine spawning.
         - spawn expression: like go, returns Future<T>; works on any fn.
         - async fn: body annotation indicating the function contains await
           expressions. Not a type distinction — any fn can be spawned.
         - await expression: suspends the current coroutine until a Future<T>
           resolves. Valid inside any function; does not require async at call
           sites when the function is executed in a goroutine context.
         - Channel<T>(capacity:) with send, receive, close, iteration.
         - select statement: multiplex over channels and timeouts; first ready
           branch runs. Optional default branch for non-blocking polls.
         - Future<T>, Future.all, Future.any, Future.race.
         - TaskGroup: structured concurrency block — all spawned tasks complete
           (or are cancelled) before the group exits; errors propagate to caller.

    2.2  Inline assembly
         - @asm[arch] blocks: x86_64, arm64 (additional arches extensible).
         - @asm[arch, feature] for ISA extensions (avx2, neon, etc.).
         - Variable name scoping: enclosing Nordvest names resolved to their
           LLVM alloca/register at the asm site.
         - @bytes[arch] for raw hex bytecode embedding.
         - Fallback: if no @asm block matches the target arch, compiler selects
           the pure-Nordvest fallback body.

    2.3  GPU acceleration
         - @gpu annotation: function compiled as a parallel kernel.
         - Target selection at compile time: CUDA (nvcc/NVVM), Metal (MSL), or
           Vulkan compute (SPIR-V). Selected via --gpu-target flag.
         - @gpu(block) for inline GPU block inside a CPU function.
         - @gpu.local for per-thread shared memory hints.
         - Runtime: data.toGPU() / data.toCPU() for explicit memory residency.
         - Automatic transfer for plain @gpu function calls from CPU code.

    2.4  C / C++ interoperability
         - @extern("lib") for C functions whose Nordvest name matches the C symbol.
         - @extern(fn: "c_symbol", lib: "lib") when the C symbol name differs from
           the Nordvest function name (e.g. the C name contains underscores or
           conflicts with a keyword).
         - @extern(cxx: "Ns::fn", lib: "lib") for C++ symbols with name demangling.
         - @CLayout annotation for C-compatible field layout and padding. Applicable
           to both struct (value type) and class (reference type); use struct for
           by-value C interop and class when passing a pointer to a C++ object.
         - @c and @cpp inline blocks (pass through to clang during compilation).
         - @include("header.h") to import a C/C++ header.
         - ptr<T> type for raw pointers (unsafe context only).

    2.5  Operator overloading
         - Define +, -, *, /, %, ^, &, |, ⊕, ==, <, >, <=, >= inside a class
           body. Bitwise ops follow same dispatch rewrite as arithmetic.
         - Seen in EXAMPLE.txt: Matrix.+(other), Matrix.*(other).
         - Compiler rewrites a + b to a.+(b) for user-defined types.

    2.6  Lazy sequences and Iterator protocol
         - Sequence<T> type: lazy, pull-based counterpart to eager [T].
         - yield statement in Sequence<T>-returning functions (generator fns).
         - Sequence<T> supports .map .filter .take .drop .zip .enumerate .toList.
         - Iterator / Iterable interfaces with associated type Item (Section 3.2).
         - for-loop desugaring via .iter() / .next() for any Iterable<T>.

    2.7  Standard library v1 (see Section 5 for API details)
         Phase 2 core modules:
         - std.io, std.fs, std.math, std.string, std.collections, std.net.http,
           std.json, std.concurrent, std.test, std.gpu, std.ffi, std.process,
           std.time, std.log, std.crypto, std.encoding, std.net, std.compress,
           std.cli, std.regex, std.rand, std.hash, std.iter, std.net.url,
           std.fmt, std.uuid, std.bytes, std.config.
         Phase 3 extended modules (see Section 3.7):
         - std.sql, std.csv, std.xml, std.toml, std.unicode, std.bignum,
           std.archive, std.html, std.stats, std.mime, std.signal,
           std.template, std.debug.

    2.8  nv CLI additions
         - nv test         run std.test-annotated test functions.
         - nv doc          generate HTML/Markdown from doc comments.
         - nv pkg          package manager (see Section 6).

  Phase 2 test targets:
    - Complete program examples (sections 20–32 of EXAMPLE.txt) compile and
      produce correct output.
    - Concurrent producer/consumer channel example runs correctly.
    - GPU addVectors produces correct result on CUDA and/or Metal target.
    - C interop: @extern sin/cos produce values matching system libm.
    - Lazy fibonacci().take(20) produces correct values without materializing
      the infinite sequence.
    - Standard library has ≥ 80% test coverage.

────────────────────────────────────────────────────────────────────────────────
  PHASE 3 — POLISH & ECOSYSTEM
  Goal: formatter, language server, package registry, perf, error messages
────────────────────────────────────────────────────────────────────────────────

Milestone: M3 "Developer Experience"

  3.1  nv fmt — canonical formatter
       - Implements the Nordvest style guide (see Section 6.4).
       - Idempotent: fmt(fmt(x)) == fmt(x) for all valid inputs.
       - Preserves blank lines and doc comments.
       - Normalizes Unicode operators to preferred forms (→ not ->, ≠ not !=,
         etc.) with a --ascii flag for terminals that cannot render Unicode.

  3.2  Language server protocol (LSP)
       - Implement nv-lsp, a standalone Language Server Protocol server backed
         by the type-checked AST from the compiler. Runs as a subprocess; all
         IDE integrations communicate with it over JSON-RPC (stdio or TCP).
       - Core LSP features (must-have for v1):
           hover type info and doc comments
           go-to-definition (including cross-module and stdlib)
           find-references and rename symbol
           completion (keywords, module members, method chains, Unicode operators)
           real-time diagnostics (type errors, null safety, exhaustiveness)
           formatting on save (delegates to nv fmt)
           code actions: introduce variable, wrap in if let, add ? propagation
       - Extended LSP features (Phase 3 stretch):
           inlay hints (inferred types on let bindings, named tuple fields)
           semantic token highlighting (richer than TextMate scopes)
           call hierarchy and type hierarchy views
           workspace symbol search
       - Target editors via LSP: Neovim (nvim-lspconfig), Zed, Helix, Emacs.
         VS Code and IntelliJ have first-class dedicated plugins (see Section 7).

  3.3  Error message quality
       - All type errors include: what was expected, what was found, and the
         source span with inline caret.
       - Null safety errors suggest if let / guard let / ?? patterns.
       - Exhaustiveness errors list the missing match cases explicitly.
       - "Did you mean?" suggestions for typos using Levenshtein distance.

  3.4  Performance work
       - Profile compile time; target < 500 ms for a 10 KLOC project.
       - Incremental compilation: cache per-module type-checked IR, recompile
         only changed modules and their dependents.
       - LLVM optimization passes: at least -O2 for nv build --release.

  3.5  Package manager (nv pkg) — see Section 6.6 for full spec.

  3.6  Documentation site
       - Static site generated from the spec/ and docs/ directories.
       - Auto-generated API docs for stdlib (from doc comments).
       - "Try Nordvest" playground (WASM-compiled interpreter or server-side).

  3.7  Standard library v2 — extended modules
       - std.sql: driver-independent SQL interface (Go database/sql model);
         bundled SQLite driver; pluggable drivers for Postgres, MySQL, etc.
       - std.csv: CSV/TSV read and write with configurable separator and quoting.
       - std.xml: DOM and SAX-style XML parsing; namespace support; error
         recovery for malformed documents.
       - std.toml: TOML v1.0 parsing and serialization; primary format for
         nv.pkg project files.
       - std.unicode: Unicode normalization (NFC/NFD/NFKC/NFKD), grapheme
         cluster iteration for emoji-correct length, codepoint properties,
         case folding.
       - std.bignum: arbitrary-precision integer (BigInt) and decimal (Decimal)
         with exact arithmetic; RoundMode for banking/finance use cases.
       - std.archive: zip and tar read/write built on std.compress streams.
       - std.html: HTML5 parsing with error tolerance, CSS selector queries
         (querySelector / querySelectorAll), and XSS sanitization.
       - std.stats: descriptive statistics — mean, median, stddev, percentile,
         histogram, Pearson and Spearman correlation.
       - std.mime: MIME type detection from extension or magic bytes; parse
         and format Content-Type headers.
       - std.signal: OS signal registration; channel-based delivery integrates
         with the async runtime for safe signal handling.
       - std.template: text templating engine with {{ }}, {% if %}, {% for %},
         {% include %}; HtmlTemplate variant auto-escapes expressions.
       - std.debug: stack traces, heap stats, simple wall-clock profiling,
         and compile-away debug assertions.

  Phase 3 test targets:
    - LSP passes all protocol conformance tests.
    - Formatter is idempotent on 100% of the test corpus.
    - Package manager can install, resolve deps, and build a real-world package.
    - All stdlib v2 modules have ≥ 80% test coverage.

────────────────────────────────────────────────────────────────────────────────
  PHASE 4 — LANGUAGE COMPLETION
  Goal: the language is fully usable; the stdlib is real; flagship programs exist
────────────────────────────────────────────────────────────────────────────────

Milestone: M4 "First Real Programs"

  Rationale: self-hosting was the original Phase 4 goal, but it is premature and
  backwards. Self-hosting is a signal of language maturity, not a cause of it.
  The stdlib is all stubs, Phase 2 codegen is incomplete, and no substantial
  program has been written in Nordvest. Trying to port the compiler before those
  problems are solved would mean implementing missing features to serve the
  compiler rather than users, and maintaining two compiler implementations in
  lockstep. The Kotlin bootstrap compiler is a solid, tested reference
  implementation — it is not a liability. Phase 4 makes the language genuinely
  usable; self-hosting is deferred to Phase 5 once that is true.

  4.1  Implement the standard library in Nordvest
       - The stdlib/std/*.nv files have signatures but no bodies. Implement:
           std.collections  HashMap<K,V>, Set<T>, BTreeMap, Queue, Stack
           std.string       trim, split, join, replace, parseFloat, parseInt, ...
           std.iter         map, filter, reduce, zip, enumerate, flatten, take, ...
           std.fs           read, write, readLines, exists, stat, walk
           std.io           stdin/stdout/stderr, buffered readers/writers
           std.math         sin/cos/tan/sqrt/log/floor/ceil/round/clamp/gcd/lcm
           std.hash         Hash trait, FNV-1a, SipHash implementations
           std.fmt          format!, sprintf-style formatting, padding, alignment
       - Each module must have ≥ 80% test coverage using nv test.
       - This is the most important deliverable: real programs need real stdlib.

  4.2  Complete stub codegen for Phase 2 features
       - async/await: emit actual coroutine frames (setjmp/longjmp or ucontext).
       - Channels: nv_chan_send / nv_chan_recv backed by a real MPSC queue.
       - Operator overloading: emit user-defined operator calls in LlvmIrEmitter.
       - Lazy sequences / generators: emit state-machine frame per yield.
       - for-in over Iterable types using the Iterator protocol.
       - Struct layout: pack fields by alignment; emit typed GEP in IR.
       - Reference counting: emit retain/release around class allocations;
         weak/unowned pointer semantics in the runtime.

  4.3  Write flagship example programs
       - Three non-trivial programs that exercise the full language:
           (a) nv-http: async HTTP/1.1 client using std.net, std.io, async/await
           (b) nv-json: streaming JSON parser returning Result<T>, using std.string
           (c) nv-bench: statistical benchmarking harness using std.stats, std.time
       - These live in examples/ and are compiled and run in CI.
       - Their purpose is to find design gaps that unit tests miss, and to serve
         as canonical demonstrations of idiomatic Nordvest.

  4.4  Harden the compiler and tooling
       - Fuzzing: feed random/mutated .nv inputs; compiler must never segfault.
       - Error message audit: every compile error has a source span and a
         human-readable suggestion; no "internal error" leaks to the user.
       - nv fmt round-trip: fmt(fmt(x)) == fmt(x) on 100% of the test corpus.
       - LSP: completion and hover tested end-to-end in VS Code and Neovim.
       - nv pkg install: resolve a real dependency graph (two packages, one
         transitive dep) and build successfully.

  Phase 4 success criteria:
    - All stdlib modules listed in 4.1 compile and pass their own test suites.
    - All three flagship programs compile and run correctly end-to-end.
    - async/await, channels, operator overloading, and lazy sequences each
      have at least one end-to-end integration test in the golden test suite.
    - 0 open "UnsupportedFeature" codegen errors on the flagship programs.
    - A new contributor can write a 100-line Nordvest program using only the
      stdlib without hitting a stub or missing feature.

────────────────────────────────────────────────────────────────────────────────
  PHASE 5 — STDLIB & PRODUCTION HARDENING
  Goal: real standard library; real reference counting; language fully usable
────────────────────────────────────────────────────────────────────────────────

Milestone: M5 "Real Programs"

  Rationale: The Kotlin bootstrap compiler is a solid, tested reference
  implementation and is retained permanently. Self-hosting is not a goal.
  Phase 5 focuses on the work that makes Nordvest genuinely usable: replacing
  stub stdlib bodies with real implementations and completing reference counting.

  Prerequisite: Phase 4 complete.

  5.1  std.string — real implementation
       - UTF-8 string as length-prefixed heap allocation in the C runtime.
       - Core operations in nv_runtime.c: slice, indexOf, contains, split,
         trim, replace, toUpper, toLower, startsWith, endsWith, toInt, toFloat.
       - String interpolation backed by real concat/format runtime.
       - stdlib/std/string.nv: real bodies via @extern + Nordvest wrappers.
       - Golden file tests for each operation; ≥ 80% test coverage.

  5.2  std.collections — real implementation
       - List<T>: growable heap array (malloc/realloc).
         Operations: add, remove, get, set, len, capacity, sort, slice, contains.
       - Map<K,V>: open-addressing hash map.
         Operations: insert, get, delete, contains, keys, values, entries.
       - Set<T>: hash set backed by Map<K,Unit>.
       - Iterator protocol wired to real for-in codegen.
       - RC integration: retain/release on element add/remove.
       - stdlib/std/collections.nv: real bodies replacing stubs.

  5.3  std.io — real implementation
       - readLine(): allocating line reader from stdin.
       - File type: open/close/read/write/seek via @extern over C stdio.
       - Buffered reader/writer wrappers in Nordvest.
       - IOError sealed class with real POSIX error codes.
       - print/println/eprintln hardened (extend existing @c stubs).

  5.4  std.fs, std.time, std.process, std.rand
       - std.fs: Path type with join/parent/stem/ext; exists/isFile/isDir/

  PHASE 7 — MULTI-FILE COMPILATION & LIBRARY LINKING
  Goal: nv build without a file arg; nv link; per-module .ll caching
────────────────────────────────────────────────────────────────────────────────

Milestone: M7 "Real Projects"

  Rationale: The bootstrap compiler (Phases 1–5) requires all application code
  to live in a single .nv file because it compiles one file at a time and emits
  all runtime helpers inline into that module. Real programs — and the flagship
  examples in examples/ — need multi-file module support. Phase 7 implements
  project-wide compilation as a distinct pipeline stage on top of the existing
  per-file compiler, without requiring changes to the type checker or codegen.

  7.1  Module collector and import graph
       - Walk the package root (directory containing nv.pkg) and discover all
         .nv files recursively.
       - Parse the module declaration from the first non-comment line of each
         file to assign it to a module. Files without a module declaration
         belong to the implicit root module (same as the package name).
       - Build a directed import graph: each module node points to the modules
         it imports. Detect and report import cycles as compile errors.
       - Produce a topological sort order; modules are compiled leaf-first so
         that every dependency's type information is available before its
         dependents are type-checked.

  7.2  Per-module incremental compilation
       - Each module (set of .nv files sharing a module declaration) is
         compiled independently to a .ll file in .nv-cache/.
       - The cache key is the SHA-256 of the concatenated source content of
         all files in the module, plus the set of imported module interface
         hashes (so downstream modules are invalidated when an API changes).
       - On a clean build every module is compiled. On an incremental build
         only modules whose cache key has changed are recompiled.
       - The existing Phase 3.4 single-file cache is unified with this system;
         nv build <file.nv> continues to work and uses the same cache layout.

  7.3  Multi-module linking via llvm-link
       - After per-module compilation, collect all .ll files (fresh + cached).
       - Invoke llvm-link to merge them into a single IR module.
       - Pass the merged module to clang to produce the final binary.
       - The inline runtime helpers (nv_print, nv_rc_retain, etc.) are emitted
         only by the root module's .ll; all other modules emit declare stubs.
         llvm-link resolves them from the single definition at link time.
         (This requires a one-time refactor of LlvmIrEmitter to separate
         "runtime definition" from "runtime declare" emission.)

  7.4  nv link subcommand
       - Expose the llvm-link + clang pipeline as a first-class nv link command
         (see Section 6.8 for the full CLI spec).
       - nv build --emit-llvm compiles a module to a standalone .ll artifact.
       - nv build --link-ll <file> injects an extra .ll into the merge step.
       - Library packages may ship .ll artifacts alongside .nv source; nv pkg
         install will compile .nv sources to .ll on install (nv build --lib
         --emit-llvm), and subsequent builds link the cached .ll directly.

  7.5  Cross-module type visibility
       - The type checker currently processes one file at a time with an
         in-memory symbol table. For multi-module builds it must be able to
         load the exported interface of a compiled dependency module without
         re-parsing its source.
       - Implement a lightweight module interface file (.nvi) written alongside
         each .ll: a JSON/TOML snapshot of the module's exported types, function
         signatures, and constants. The type checker reads .nvi files for
         imported modules rather than re-compiling them from source.
       - Format: human-readable; versioned with a schema version field so
         interface evolution is detectable and old caches are invalidated.

  Phase 7 success criteria:
    - nv build (no file arg) from a project root compiles all .nv files and

    - examples/nv-http, examples/nv-json, examples/nv-bench are split into

    - Incremental rebuild after touching one file recompiles only the affected
      module and its dependents.
    - nv link foo.ll bar.ll -o myapp produces a working binary.
    - A library package can be pre-compiled to .ll (nv build --lib --emit-llvm)

================================================================================
  3. FORMAL SYNTAX SPECIFICATION
================================================================================

The formal syntax specification lives at spec/grammar.peg (PEG grammar) and
spec/language.md (narrative spec). Together they are the normative reference.

────────────────────────────────────────────────────────────────────────────────
  3.1  Grammar Format Choice
────────────────────────────────────────────────────────────────────────────────

Use PEG (Parsing Expression Grammar) rather than EBNF for the following reasons:

  - PEG is unambiguous by construction (ordered choice /); EBNF requires
    separate disambiguation rules.
  - PEG scannerless parsers handle indentation sensitivity cleanly via
    parameterized rules.
  - PEG tools (pest, peg.js, Ohm, or a hand-written recursive-descent parser
    following the PEG structure) are widely available.

The grammar file (spec/grammar.peg) is normative. The Kotlin parser in
compiler/ implements this grammar exactly; deviations are bugs.

────────────────────────────────────────────────────────────────────────────────
  3.2  Grammar Document Structure
────────────────────────────────────────────────────────────────────────────────

spec/grammar.peg is organized into the following sections:

  SECTION A — Lexical Rules (tokens)
    A.1  Whitespace and line endings
         - NEWLINE := '\n' / '\r\n' / '\r'
         - BLANK_LINE, COMMENT, ignored between top-level items.
    A.2  Indentation tokens
         - INDENT: increase in leading spaces relative to enclosing block.
           The grammar is parameterized by an explicit indent level (indented(n))
           so that block-level rules receive their expected indent depth.
         - DEDENT: return to previous indent level.
         - Rules for tab expansion (one tab = 4 spaces; tabs and spaces must
           not be mixed within a file).
    A.3  Integer and float literals
         - Decimal, hex (0x), binary (0b), octal (0o).
         - Underscores as digit separators: 1_000_000.
         - Float: digit+ '.' digit+ (exponent)? e.g. 1.5e-3.
    A.4  String literals
         - Double-quoted strings with escape sequences (\n, \t, \\, \", \uXXXX).
         - Interpolation: '{' expr '}' and '{' expr ':' format_spec '}'.
         - Raw strings (no escape processing): r"..." (no escapes, no interpolation);
           r"""...""" multi-line with indentation stripped to the level of the closing """.
    A.5  Identifiers
         - Start: Unicode letter or '_'.
         - Continue: Unicode letter, digit, or '_'.
         - Greek letters (π, μ, σ) are valid identifiers.
         - Keywords are a subset of identifiers; listed exhaustively.
    A.6  Unicode operator tokens
         - → ∀ ∃ ∑ ∏ ∈ ∧ ∨ ¬ π e ∞ ≤ ≥ ≠ ⊕
         - Also accepted: -> && || ! <= >= != xor (ASCII aliases).
         - The pipeline operator |> is two ASCII characters forming one token.
           The single | is the bitwise-OR token. Lexer uses maximal munch: |>
           wins when > immediately follows |.
         - The power operator ^ is a single ASCII character (NOT bitwise XOR).
         - Bitwise tokens: & | ~ << >> (single-char or two-char, not composite).
           Compound assignment variants: &= |= ⊕= <<= >>=.
         - Integer division: the sole token is ÷ (U+00F7). The two-slash //
           sequence is ALWAYS lexed as a line comment regardless of position.
           There is no // integer-division operator in Nordvest.
         - xor is a reserved keyword acting as an infix operator (alias for ⊕).
    A.7  Annotation tokens
         - @gpu @gpu(block) @gpu.local @cpu @asm @bytes @extern @CLayout
           @c @cpp @include
         - @if @else  (compile-time conditional; predicate follows in parentheses)
         - @derive @builder @newtype @lazy  (structural-boilerplate annotations;
           see Chapter 17)
         - @label identifiers on loops are not separate annotation tokens; the
           grammar parses '@' identifier immediately before for/while keywords.

  SECTION B — Expression Grammar
    B.1  Operator precedence table (low to high):
         Level 1  (lowest)  lambda       x → expr
         Level 2            pipeline     expr |> expr
         Level 3            or           expr ∨ expr  (∨ / ||)
         Level 4            and          expr ∧ expr  (∧ / &&)
         Level 5            not          ¬expr  (¬ / !)
         Level 6            comparison   expr (== ≠ < > ≤ ≥) expr
         Level 6.5          bitwise-or   expr | expr
         Level 6.6          bitwise-xor  expr ⊕ expr  (⊕ / xor)
         Level 6.7          bitwise-and  expr & expr
         Level 6.8          bit-shift    expr (<< | >>) expr
         Level 6.9          bitwise-not  ~expr  (unary)
         Level 7            addition     expr (+ -) expr
         Level 8            multiply     expr (* / ÷ %) expr
         Level 9            power        expr ^ expr  (right-associative)
         Level 10           unary        -expr
         Level 11           postfix      expr? expr! expr.member expr[...] call
                                          expr is type   expr as? type   expr as! type
         Level 12 (highest) primary      literal, identifier, (expr), [expr,...],
                                          quantifier, match, if-expr
    B.2  Quantifier expressions
         - ∀ binding ∈ expr ':' expr      (boolean — true if pred holds for all)
         - ∃ binding ∈ expr ':' expr      (boolean — true if pred holds for any)
         - ∑ binding ∈ expr ':' expr      (numeric — sum of expr over collection)
         - ∏ binding ∈ expr ':' expr      (numeric — product of expr over collection)
         - ∑ expr                         (numeric, no binding — sum of iterable)
         - ∏ expr                         (numeric, no binding — product of iterable)
         - ∀ binding ∈ expr '→' block     (imperative loop — desugars to for-loop;
                                           preferred in mathematical/scientific code)
         - ∃ binding ∈ expr ':' pred '→' block  (imperative first-match — executes
                                           body for first element where pred is true)
    B.3  Range expressions
         - '[' expr ',' expr ']'  closed
         - ']' expr ',' expr '['  open
         - '[' expr ',' expr '['  half-open left-closed
         - ']' expr ',' expr ']'  half-open right-closed
    B.4  List comprehension
         - '[' expr 'for' binding 'in' expr (',' binding 'in' expr)* ('if' expr)? ']'
         - Also: '[' expr 'for' binding '∈' expr ... ']'
    B.5  Lambda
         - binding '→' expr                  (single-param, no parens needed)
         - '(' params ')' '→' expr           (multi-param or zero-param)
         - '(' ')' '→' expr                  (zero-param — void input)
         Parameter types are inferred from the surrounding call's expected
         fn type; explicit annotation allowed: (x: int) → x * 2.
         Trailing lambda syntax: a lambda that is the last argument to a
         call may appear outside the parens as a { } block:
           call(a, b) { x → expr }           (inline)
           call { x → expr }                 (only argument, parens omitted)
           call { x →                        (multi-line; } closes the block)
               stmt1
               stmt2 }
         The { } delimiter is used exclusively for trailing lambdas; it does
         not appear elsewhere in the grammar. nv fmt enforces consistent style
         (trailing form preferred when lambda is the sole or last argument).
    B.6  Match expression
         - 'match' expr NEWLINE INDENT arm+ DEDENT
         - arm := pattern (':' / ',') → expr  /  pattern ':' NEWLINE INDENT stmts DEDENT
    B.7  If expression
         - Statement form (multi-line):
           'if' expr NEWLINE INDENT stmts DEDENT ('else' NEWLINE INDENT stmts DEDENT)?
         - Inline expression form:
           'if' expr 'then' expr ('else' 'if' expr 'then' expr)* 'else' expr
           The 'then' keyword is the disambiguator: its presence signals an inline
           conditional expression rather than a statement. 'then' is only valid in
           this position and must always be followed by an else clause. There is no
           ternary operator (?:) — this inline form serves the same purpose without
           the ? disambiguation complexity.
    B.8  String interpolation (handled during parsing, not in lexer)
         - Every '{' inside a string literal starts an interpolation sub-parse.
    B.9  Yield expression
         - 'yield' expr  — only valid as a statement inside a Sequence<T>-returning
           function. Transforms the function into a generator (coroutine).
         - A function with at least one yield statement must declare return type
           Sequence<T>; the compiler infers T from the yielded expression types.
    B.10 Cast and type-test expressions
         - expr 'is' type       — boolean type check with smart cast: in the
                                  then-branch of an if-statement, the variable is
                                  narrowed to the tested type if the compiler can
                                  prove it is not reassigned between the is-check
                                  and the use site.
         - expr 'as?' type      — explicit safe downcast; result type is type?
                                  Use when the smart cast cannot apply (e.g., the
                                  narrowed value is stored in a new binding).
         - expr 'as!' type      — forced downcast; panics at runtime if wrong type

  SECTION C — Statement Grammar
    C.1  Variable declarations
         - 'let' identifier (':' type)? ('=' expr)?
         - 'var' identifier (':' type)? ('=' expr)?
    C.2  Assignment
         - lvalue '=' expr
         - lvalue ('+=' '-=' '*=' '/=' '&=' '|=' '⊕=' '<<=' '>>=') expr
    C.3  Function declaration
         - 'fn' identifier ('<' type_params '>')? '(' params ')' ('→' type)? ('throws' type)?
           NEWLINE INDENT stmts DEDENT
         - 'throws' E is syntactic sugar for a Result<T, E> return type, where T is
           inferred from the function's return statements. A throws fn can be called
           with ? propagation in any context that expects Result<_, E>. The two forms
           interoperate: Result<T, E> and throws E are the same at the IR level.
    C.4  Async function
         - 'async' 'fn' ...  (same as fn but with compile-time enforcement:
           'await' expressions are only valid inside the body of an 'async fn';
           the compiler rejects 'await' in a non-async fn with an error.
           async fn is not a distinct callable type — any fn can be spawned;
           the async annotation purely triggers the coroutine state-machine
           transformation and the await-validity check.)
    C.5  Return statement
         - '→' expr          (preferred)
         - 'return' expr
    C.6  For loop
         - ('@' label)? 'for' binding 'in' expr NEWLINE INDENT stmts DEDENT
         - 'for' '(' binding ',' binding ')' 'in' expr ... (tuple destructuring)
         - Desugars (for Iterable<T>): let _it = expr.iter(); while let _v = _it.next()
    C.7  While loop
         - ('@' label)? 'while' expr NEWLINE INDENT stmts DEDENT
    C.8  Break and continue
         - 'break' ('@' label)?
         - 'continue' ('@' label)?
    C.9  Go statement (fire-and-forget goroutine)
         - 'go' ('[' capture_list ']')? (call_expr | block)
         - capture_list := ('copy')? identifier (',' ('copy')? identifier)*
         - Desugars to: concurrent.spawn(fn() { ... }) discarding the Future
    C.10 Spawn expression (goroutine with awaitable result)
         - 'spawn' (call_expr | block)
         - Type: Future<T> where T is the return type of the spawned expression
         - Works on any fn, async or not; async marks presence of await in body
         - Desugars to: concurrent.spawn(fn() → T { ... })
    C.11 Select statement
         - 'select' NEWLINE INDENT select_arm+ DEDENT
         - select_arm :=
             binding 'from' expr ':' stmt         // channel receive
           | '_' 'from' expr ':' stmt             // receive, discard value
           | 'after' duration ':' stmt            // timeout arm
           | 'default' ':' stmt                   // non-blocking poll arm
         - duration := int_lit ('ms' | 's' | 'm' | 'h')
         - Semantics: evaluates all channel exprs, blocks until one is ready;
           if multiple ready simultaneously, one is chosen pseudo-randomly.
           default arm (if present) runs immediately if no channel is ready.
         - At most one default arm and one after arm per select.
    C.12 Try/catch
         - 'try' NEWLINE INDENT stmts DEDENT
           ('catch' binding ':' type NEWLINE INDENT stmts DEDENT)*
           ('finally' NEWLINE INDENT stmts DEDENT)?
    C.13 Defer statement
         - 'defer' stmt
         - 'defer' NEWLINE INDENT stmts DEDENT  (block form)
         - Semantics: the deferred code is registered at the defer site and runs
           when the enclosing function scope exits, in LIFO order.
    C.15 Unsafe block
         - 'unsafe' NEWLINE INDENT stmts DEDENT
         - Grants permission to: dereference ptr<T>, perform raw pointer arithmetic,
           call @extern functions with ptr parameters without an enclosing @c block.
         - The compiler tracks unsafe blocks; using ptr<T> outside an unsafe block
           or @c/@cpp block is a compile error.
         - @c and @cpp blocks implicitly provide an unsafe context.
    C.16 Async/await rules (compile-time enforced)
         - 'await' expr is only valid inside an 'async fn' body.
         - An 'async fn' called from a non-async context must be called via
           'spawn' (returns Future<T>) or 'go' (fire-and-forget).
         - The compiler rejects 'await' outside an async fn with a clear error.
    C.14 Conditional compilation
         - '@if' '(' cc_pred ')' NEWLINE decls ('@else' NEWLINE decls)?
         - cc_pred := 'platform' '==' str_lit
                    | 'arch' '==' str_lit
                    | 'debug' | 'release'
                    | 'feature' '(' str_lit ')'
         - Applies at declaration scope only (top-level, class body, module level).
           Not valid inside expression or statement context.

  SECTION D — Declaration Grammar
    D.1  Struct declaration (value type)
         - 'struct' identifier ('<' type_params '>')? '(' constructor_params ')'
           ('where' constraints)?
           NEWLINE INDENT struct_members DEDENT
         - struct_members: fn declarations and computed properties only.
           No inheritance (no : base_type). No body field initializers (all state
           comes from constructor params). Fields are mutable (var semantics).
         - Semantics: value type. Assignment copies all fields. No reference counting.
           The compiler lays out structs inline (no heap allocation) unless the size
           exceeds a platform threshold, after which it may box automatically.
    D.2  Class declaration
         - 'class' identifier ('<' type_params '>')? ('(' constructor_params ')')?
           (':' base_type (',' interface_type)*)? ('where' constraints)?
           NEWLINE INDENT class_members DEDENT
         - class_members may include field declarations with ownership modifiers:
             field_decl := ('weak' | 'unowned')? ('var' | 'let') identifier ':' type ('=' expr)?
           weak var f: T?   — nullable; zeroed automatically when referent is freed
           unowned let f: T — non-nullable; accessing after free is undefined (crash)
    D.3  Sealed class
         - 'sealed' 'class' identifier NEWLINE INDENT variant+ DEDENT
         - variant := identifier ('(' params ')')? NEWLINE
    D.4  Interface
         - 'interface' identifier ('<' type_params '>')? NEWLINE INDENT interface_members DEDENT
         - interface_members := method_sig | associated_type_decl | default_impl
    D.4b Associated type declaration (inside interface body)
         - 'type' identifier           (abstract associated type)
    D.4c Associated type binding (inside class/record/struct body implementing the interface)
         - 'type' identifier '=' type  (concrete binding)
    D.4d Associated type constraint (in where clause)
         - 'where' T '.' identifier (':' constraint | '==' type)
    D.5  Record
         - 'record' identifier '(' params ')' NEWLINE
         - Immutable reference type. All fields are let. Auto-generates ==, str(), hash().
    D.6  Enum
         - 'enum' identifier (':' type)? NEWLINE INDENT case_list DEDENT
    D.6  Extension
         - 'extend' type (':' interface_type (',' interface_type)*)? ('where' constraints)? NEWLINE INDENT fn_decls DEDENT
         - The optional ': interface_type' list is retroactive conformance: declares
           that 'type' satisfies those interfaces. All interface methods must be
           implemented in the extend block (unless already present on the type).
           Conformance is nominal and module-scoped. Conflicting conformances for
           the same type+interface pair across imported modules are a compile error.
    D.7  Type alias
         - 'type' identifier '=' type
         - Semantics: transparent (structural) alias. The new name and the aliased
           type are fully interchangeable; no implicit conversion is needed, and no
           new type-checking boundary is introduced. Example: given
           'type Meters = float', a float literal 42.0 is valid where Meters is
           expected and vice-versa. For a nominal (distinct) wrapper type, use a
           single-field struct or record instead.
    D.8  Module and imports
         - 'module' qualified_name
         - 'import' qualified_name ('as' identifier)?
         - 'import' qualified_name '.' identifier
         - 'pub' 'import' qualified_name   (re-export; valid only in mod.nv)
           Semantics: makes the named sub-module visible to consumers of this
           package. A consumer can write 'import parent.submodule' and access its
           pub symbols as 'submodule.Symbol'. Symbols are NOT merged into the
           parent module's namespace; only the sub-module namespace is exposed.
    D.9  Visibility modifiers
         - 'pub'               exported from the package
         - 'pub' '(' 'pkg' ')' visible to all modules in this package, not exported
         - (none)              file-private (visible in this .nv file only)
    D.10 Computed property
         - 'fn' identifier ('→' type)?   (no parameter list — property, not method)
           NEWLINE INDENT stmts DEDENT
         - With explicit get/set:
           'fn' identifier '→' type NEWLINE INDENT
               ('get' ('→' expr | NEWLINE INDENT stmts DEDENT))
               ('set' '(' identifier ')' (NEWLINE INDENT stmts DEDENT | stmt))?
           DEDENT
    D.11 Subscript operators (inside class/record/extend body)
         - 'fn' '[' ']' '(' params ')' '→' type NEWLINE INDENT stmts DEDENT
         - 'fn' '[' ']' '=' '(' params ',' identifier ':' type ')' NEWLINE INDENT stmts DEDENT
    D.12 Multi-file module resolution
         - All .nv files in directory dir/ declaring 'module foo.bar' form one module.
         - The module name must match the directory path relative to the package root.
         - Subdirectory dir/sub/ forms module foo.bar.sub automatically.
         - A file named mod.nv in dir/ may re-export sub-modules with pub import.

  SECTION E — Type Grammar
    E.1  Named types: identifier, qualified (a.b.c), generic (T<U, V>)
    E.2  Nullable: type '?'
    E.3  Array: '[' type ']'        — e.g. [int], [str]
    E.3b Map: '[' type ':' type ']' — e.g. [str: int]; literal syntax [k: v, ...]
         Empty map literal: '[' ':' ']'  (i.e. [:])
         Empty array literal: '[' ']'   (i.e. [])
         Disambiguated in expression position: [expr, ...] is array; [expr: expr, ...]
         is map. In type position: [T] is array; [K: V] is map.
    E.4  Matrix: '[[' type ']]'
    E.5  Function type: 'fn' '(' types ')' '→' type
         - Zero params:  fn() → ()          (void callback)
         - One param:    fn(int) → str
         - Multi-param:  fn(int, int) → bool
         - Void return () may also be written as the absence of a return type
           annotation in a fn declaration, but in type position () is explicit.
         - fn types are first-class values: storable in let/var, passable as
           arguments, returnable from functions. No implicit currying.
    E.6  Tuple type: '(' type (',' type)+ ')'
    E.7  Named tuple type: '(' (identifier ':' type)+ ')'
    E.8  Result: 'Result' '<' type (',' type)? '>'
    E.9  Pointer (unsafe): 'ptr' '<' type '>'
    E.10 Lazy sequence: 'Sequence' '<' type '>'
         - A Sequence<T> is a lazy, pull-based stream. Returned by generator fns
           (those with yield statements). Supports .map .filter .take .drop .zip
           .enumerate .toList; all operations are lazy except .toList / for loops.

  SECTION F — Pattern Grammar (for match arms)
    F.1  Literal patterns: int, float, bool, str, nil
    F.2  Range patterns: range notation ([a,b], etc.)
    F.3  Type patterns: ClassName '(' binding_list ')'  (sealed variants)
         - Positional:   Dog(name)
         - Named fields: Circle(radius: r)
         - Nested:       Ok(Some(x))  — patterns compose recursively
    F.4  Wildcard: '_'
    F.5  Binding: 'let' identifier  (binds and names a value in the arm)
    F.6  Tuple pattern: '(' pattern (',' pattern)+ ')'
    F.7  Guard: pattern 'if' expr
    F.8  Multiple patterns: pattern (',' pattern)+   (comma — same arm, same body)
    F.9  Or-patterns: pattern '|' pattern            (either variant → same body)
         - Example: Add(_) | Mul(_): → "binary"
         - All alternatives must bind the same set of names at the same types.
    F.10 Recursive patterns: sealed types may reference themselves; the pattern
         grammar is therefore recursive. The exhaustiveness checker must handle
         recursive constructors without infinite expansion (use depth-bounded
         unfolding or set-based coverage analysis).

────────────────────────────────────────────────────────────────────────────────
  3.3  Indentation Sensitivity Rules
────────────────────────────────────────────────────────────────────────────────

Nordvest uses significant indentation following these rules:

  - The canonical indent unit is 4 spaces. The formatter enforces this.
  - Tabs are forbidden in new code; the formatter converts them.
  - A block begins after a colon-terminated line or after a declaration header
    line (fn, class, if, for, while, match, etc.) — the next non-blank line
    must be indented relative to the header.
  - A block ends when a non-blank line is indented less than or equal to the
    block's opener.
  - Continuation lines (pipeline |>, multi-line array literals, multi-line
    argument lists) may be indented at any level strictly greater than the
    statement's starting column. The parser tracks "expression continuation"
    vs. "new statement" based on whether the current expression is complete.
  - One-liner sugar: a single-expression body may appear on the same line as
    the header after an indented block would normally start, separated by the
    body expression directly (no colon): "fn area() → π * r^2"

  5. STANDARD LIBRARY DESIGN
================================================================================

All standard library modules are written in Nordvest (Phase 2+), using @extern
and @c/@cpp for thin OS-interface wrappers where necessary. Module names follow
the qualified path convention (std.io, std.collections.HashMap, etc.).

  5.1  std.io — Input/Output
────────────────────────────────────────────────────────────────────────────────

  Purpose: stdin/stdout/stderr, formatted printing, basic file reading.

  // Console output
  fn print(value: str)                    // println with newline
  fn print(value: str, end: str = "\n")   // explicit line ending
  fn eprint(value: str)                   // stderr
  fn printf(format: str, values: any...)  // printf-style formatting

  // Console input
  fn readLine() → str?                    // nil on EOF
  fn readLines() → [str]                  // all lines until EOF
  fn readInt() → Result<int>
  fn readFloat() → Result<float>

  // File (basic, see std.fs for full FS operations)
  class File
    pub static fn read(path: str) → str throws IOError
    pub static fn readLines(path: str) → [str] throws IOError
    pub static fn write(path: str, content: str) throws IOError
    pub static fn append(path: str, content: str) throws IOError

  // Buffered streams
  class BufferedReader(stream: InputStream)
    pub fn readLine() → str?
    pub fn readAll() → str
    pub fn lines() → [str]           // lazy iteration

  class BufferedWriter(stream: OutputStream)
    pub fn write(s: str)
    pub fn writeLine(s: str)
    pub fn flush()

  // Standard streams
  pub let stdin:  BufferedReader
  pub let stdout: BufferedWriter
  pub let stderr: BufferedWriter

  // Error type
  class IOError(message: str) : Error
    pub let message: str
    pub let code: int?                // OS error code, if available

  5.8  std.concurrent — Async Runtime, Channels, Futures
────────────────────────────────────────────────────────────────────────────────

  Purpose: the async runtime, synchronization primitives, and concurrency
  building blocks. The runtime is an M:N green-thread scheduler; channel sends
  and I/O automatically suspend the calling coroutine — no OS thread blocks.

  // Channel<T> — buffered or unbuffered message passing
  class Channel<T>(capacity: int = 0)   // 0 = unbuffered (rendezvous)
    pub fn send(value: T)               // suspends coroutine if full
    pub fn trySend(value: T) → bool     // non-blocking; returns false if full
    pub fn receive() → T?               // suspends until value available; nil when closed
    pub fn tryReceive() → T?            // non-blocking; nil if empty or closed
    pub fn close()
    pub fn isClosed() → bool
    // Channel is iterable: "for v in ch" iterates until closed

  // Future<T>
  class Future<T>
    pub static fn value(v: T) → Future<T>         // immediately resolved
    pub static fn all(futures: [Future<T>]) → Future<[T]>
    pub static fn any(futures: [Future<T>]) → Future<T>
    pub static fn race(futures: [Future<T>]) → Future<T>  // first to resolve
    pub fn map<U>(f: fn(T) → U) → Future<U>
    pub fn flatMap<U>(f: fn(T) → Future<U>) → Future<U>
    pub fn onSuccess(f: fn(T))
    pub fn onFailure(f: fn(Error))
    pub fn cancel()                                // request cancellation
    pub fn await() → T                            // suspend coroutine; use with await keyword

  // TaskGroup — structured concurrency
  // All tasks spawned inside the group must complete before the group resolves.
  // If any task returns an error or panics, the remaining tasks are cancelled
  // and the error propagates to the caller.
  class TaskGroup
    pub fn spawn(f: fn() → T)    // add a task to the group

  // "await TaskGroup" desugars to: let _g = TaskGroup(); ...; await _g.join()

  // Mutex and RWLock
  class Mutex<T>(value: T)
    pub fn lock(action: fn(T) → U) → U    // acquire, run, release
    pub fn tryLock(action: fn(T) → U) → U?

  class RWLock<T>(value: T)
    pub fn read(action: fn(T) → U) → U
    pub fn write(action: fn(T) → U) → U

  // Atomic primitives
  class Atomic<T>(value: T) where T: Numeric
    pub fn load() → T
    pub fn store(value: T)
    pub fn fetchAdd(delta: T) → T
    pub fn fetchSub(delta: T) → T
    pub fn compareExchange(expected: T, desired: T) → bool

  // Semaphore
  class Semaphore(permits: int)
    pub fn acquire()
    pub fn release()
    pub fn tryAcquire() → bool

  // Runtime control
  fn concurrent.spawn<T>(f: fn() → T) → Future<T>   // any fn, async or not
  fn concurrent.sleep(ms: int)                       // suspend for duration
  fn concurrent.yield()                              // voluntarily yield coroutine

  // Desugaring reference:
  //   go expr           →  concurrent.spawn(fn() { expr })          // Future discarded
  //   spawn expr        →  concurrent.spawn(fn() → T { expr })      // Future returned
  //   await TaskGroup { ... }  →  structured cancellation scope

  5.24  std.net.url — URL Parsing and Building

  record Url

    pub let port: int?               // nil if not explicit
    pub let path: str                // always starts with "/" for http(s)

      defer db.close()
      db.execute("CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT)")?
      db.execute("INSERT INTO users VALUES (?, ?)", 1, "Alice")?

          rows.scan(&id, &name)?

  5.41  std.config — Configuration System
────────────────────────────────────────────────────────────────────────────────

  Purpose: type-safe configuration binding from TOML files and environment
  variables, with profile support, optional hot-reload, and compile-time
  binding generation via the @config and @env annotations.

  Source priority (highest wins):
    1. Environment variables
    2. CLI overrides   (--config.<key>=<val> passed to the program)
    3. config.local.toml   (local developer overrides; gitignored by nv new)
    4. config.{profile}.toml  (e.g. config.prod.toml, config.test.toml)
    5. config.toml             (project defaults; committed to source control)
    6. Struct field defaults   (code-level fallback)

  Profile activation:
    - NV_ENV environment variable (NV_ENV=prod)
    - --env=<profile> CLI flag passed to nv run / nv build

  Key-name normalisation:
    Sources use different naming conventions; the binding layer normalises all
    three forms to the same logical key before merging:
      TOML key:     snake_case or kebab-case  (max_conns, max-conns)
      Env var:      UPPER_SNAKE_CASE          (DATABASE_MAX_CONNS)
      Struct field: camelCase                 (maxConns)

  ── ConfigLoader ──────────────────────────────────────────────────────────────

  class ConfigLoader
    pub static fn default() → ConfigLoader
        // Standard stack: config.toml → config.{NV_ENV}.toml →
        // config.local.toml → env vars
    pub fn fromFile(path: str) → ConfigLoader
    pub fn fromEnv(prefix: str? = nil) → ConfigLoader
    pub fn fromArgs(args: [str]) → ConfigLoader    // --key=val pairs
    pub fn withProfile(name: str) → ConfigLoader
    pub fn withDefaults(table: [str: ConfigValue]) → ConfigLoader
    pub fn load() throws IOError → Result<Config>

  ── Config ────────────────────────────────────────────────────────────────────

  class Config
    pub fn [](key: str) → ConfigValue?       // dot-path: cfg["server.port"]
    pub fn bind<T>(prefix: str = "") → Result<T>
        // Calls T.load(from: self) — generated by @config.
        // Calls validate() on the bound value if the method exists.
    pub fn merge(other: Config) → Config     // self wins on conflicts
    pub fn profile() → str?                  // active profile name
    pub fn onReload(handler: fn(Config) → ())
        // Registers a callback fired when any watched source file changes.
        // Only active for fields annotated @config(reload: true).

  ── ConfigValue ───────────────────────────────────────────────────────────────

  sealed class ConfigValue
    Str(value: str)
    Int(value: int64)
    Float(value: float64)
    Bool(value: bool)
    Duration(value: time.Duration)   // parsed from "30s", "1m30s", "500ms"
    Array(items: [ConfigValue])
    Table(entries: [str: ConfigValue])

  extend ConfigValue
    pub fn str()      → str?
    pub fn int()      → int64?
    pub fn float()    → float64?
    pub fn bool()     → bool?
    pub fn duration() → time.Duration?
    pub fn [](key: str) → ConfigValue?   // subscript into Table variant
    pub fn [](i: int)   → ConfigValue?   // subscript into Array variant

  ── ConfigError ───────────────────────────────────────────────────────────────

  sealed class ConfigError
    MissingField(path: str)
    TypeMismatch(path: str, expected: str, got: str)
    ParseError(source: str, message: str)
    ValidationFailed(message: str)

  ── @config annotation ────────────────────────────────────────────────────────

  Syntax:  @config("prefix")
           @config("prefix", reload: true)

  Applied to struct, record, or class. At compile time, generates:
    - static fn load(from cfg: Config) → Result<T>
      Extracts each field from the subtree at "prefix", applies type coercion,
      uses struct field defaults when keys are absent, and returns
      Err(ConfigError.missingField(path)) for required fields that are missing.
    - If T defines fn validate() → Result<()>, load() calls it after binding.

  reload: true — the ConfigLoader watches the source files for changes and
  fires onReload callbacks. The struct itself is immutable per-snapshot; callers
  must hold a var binding and reassign on reload.

  ── @env field annotation ─────────────────────────────────────────────────────

  Syntax:  @env("VAR_NAME")
           @env("VAR_NAME", sensitive: true)

  Applies to a constructor parameter inside a @config struct. Binds the named
  environment variable to the field; the env var wins over the TOML file value
  (consistent with overall priority order).

  sensitive: true — the value is masked as "***" in:
    - @derive(Show) output
    - std.log formatted output
    - debug.stackTrace() variable dumps
  The compiler enforces this; leaking the value requires an explicit .value cast.

  Auto-derived env name (when @env is absent):
    UPPER_SNAKE(prefix) + "_" + UPPER_SNAKE(fieldName)
    e.g.  @config("database") + maxConns  →  DATABASE_MAX_CONNS

  ── Example ───────────────────────────────────────────────────────────────────

  import std.config

  @config("server")
  struct ServerConfig(
      host:    str      = "0.0.0.0",
      port:    int      = 8080,
      timeout: Duration = Duration.seconds(30),
      tls:     bool     = false
  )
      fn validate() → Result<()>
          if self.tls and self.port == 80
              → Err(ConfigError.validationFailed("TLS enabled but port is 80"))
          → Ok(())

  @config("database")
  struct DatabaseConfig(
      @env("DATABASE_URL")
      url: str,

      @env("DB_MAX_CONNECTIONS")
      maxConns: int = 20,

      @env("DB_PASSWORD", sensitive: true)
      password: str? = nil
  )

  fn main() throws
      let cfg    = ConfigLoader.default().load()?
      let server = cfg.bind<ServerConfig>()?
      let db     = cfg.bind<DatabaseConfig>()?
      print("Listening on {server.host}:{server.port}")
      startServer(server, db)


================================================================================
  6. TOOLING — THE nv CLI
================================================================================

The "nv" binary is the single entry point for all language tooling. It is
written in Kotlin for the bootstrap (Phase 1), then rewritten in Nordvest
during Phase 4.

────────────────────────────────────────────────────────────────────────────────
  6.1  nv run — Compile and Execute
────────────────────────────────────────────────────────────────────────────────

  Usage:  nv run [options] <file.nv> [-- <program args>]

  Behavior:
    - Compiles <file.nv> and all transitive imports to a temporary binary.
    - Executes the binary, passing any arguments after -- to the program.
    - The temporary binary is cached by content hash; re-running an unchanged
      file does not recompile.

  Options:
    --release           Enable optimizations (LLVM -O2). Default: debug mode.
    --target <triple>   Cross-compile target (e.g. aarch64-linux-gnu).
    --gpu-target <t>    GPU backend: cuda | metal | vulkan. Default: auto-detect.
    --no-cache          Force recompile.
    --verbose           Print compilation steps.
    --emit-llvm         Print LLVM IR to stdout and exit (don't compile to binary).

────────────────────────────────────────────────────────────────────────────────
  6.2  nv build — Compile to Binary
────────────────────────────────────────────────────────────────────────────────

  Usage:
    nv build [options] <file.nv>     single-file build (Phase 1–5, current)
    nv build [options]               project build     (Phase 7, planned)

  Single-file behavior (current):
    - Compiles <file.nv> and all transitive stdlib imports to a native binary.
    - Default output name: stem of input file.
    - Limitation: cross-file user imports are not resolved; all application
      code must live in one file. Multi-file project builds are Phase 7.

  Project build behavior (Phase 7, planned):
    - Walks up from the current directory to find nv.pkg; errors if not found.
    - Collects all .nv files in the package root and sub-modules (subdirectories
      sharing a module declaration hierarchy). Files sharing the same module
      declaration form one module; subdirectories are sub-modules.
    - Resolves the full import graph across all modules; reports cycle errors.
    - Compiles each module to .ll in .nv-cache/; skips modules whose source
      hash is unchanged (incremental, reusing Phase 3.4 per-file cache).
    - Merges all .ll outputs via llvm-link into a single IR module, then passes
      the merged module to clang to produce the final native binary.
    - Entry point: the unique fn main() found among all compiled files.
    - Default output name: name field from nv.pkg; falls back to directory name.
    - Sub-module .ll files may optionally be cached as pre-linked .ll libraries
      (see nv link, section 6.8) for faster incremental rebuilds.

  Options (both forms):
    -o <output>         Output binary name.
    --release           Optimized build (-O2, strip debug info).
    --debug             Debug build with DWARF info and sanitizers.
    --target <triple>   Cross-compile.
    --gpu-target <t>    GPU backend.
    --static            Statically link the runtime.
    --emit-llvm         Emit merged .ll file instead of binary.
    --emit-asm          Emit .s assembly instead of binary.
    --lib               Compile as a shared library (.so / .dylib / .dll).
    --link-ll <file>    Link an additional pre-compiled .ll file into the build.
                        May be repeated. Applied after module compilation.

────────────────────────────────────────────────────────────────────────────────
  6.3  nv test — Run Tests
────────────────────────────────────────────────────────────────────────────────

  Usage:  nv test [options] [<path>]

  Behavior:
    - Discovers all files matching *_test.nv and any .nv file containing @test
      annotations in <path> (default: current directory, recursive).
    - Compiles all test functions into a single test binary.
    - Runs the binary; reports pass/fail/skip per test.

  Options:
    --filter <pattern>  Run only tests whose name matches pattern (glob or regex).
    --format <fmt>      Output format: tap (default), junit, json, pretty.
    --bench             Also run @bench benchmarks; report ns/op.
    --timeout <ms>      Timeout per test (default: 5000 ms).
    --parallel <n>      Run tests in parallel with n workers (default: 1).
    --coverage          Generate coverage report (lcov format).
    --no-fail-fast      Continue running after first failure.

────────────────────────────────────────────────────────────────────────────────
  6.4  nv fmt — Format Source Code
────────────────────────────────────────────────────────────────────────────────

  Usage:  nv fmt [options] [<file.nv>...]

  Behavior:
    - Reformats each file in place (or to stdout with --stdout).
    - If no files are given, formats all .nv files in the current directory
      recursively.
    - Exits with code 0 if no changes were made; code 1 otherwise (useful for
      CI "format check" step with --check).

  Style rules enforced:
    - 4-space indentation, no tabs.
    - Prefer Unicode operators: → over ->, ≠ over !=, ≤ over <=, ≥ over >=,
      ∧ over &&, ∨ over ||, ¬ over !, ÷ for integer division,
      ⊕ over xor. (Override with --ascii to keep ASCII forms.)
    - No ternary operator (?:) — use inline if/then/else instead.
    - One blank line between top-level declarations.
    - Two blank lines between sections delimited by ─── comments.
    - Trailing whitespace removed.
    - Align consecutive let/var declarations when they share context.
    - Normalize string quote style (always double-quoted).
    - Long lines wrapped at 100 columns.

  Options:
    --check             Dry run; exit 1 if any file would change.
    --stdout            Write formatted output to stdout, not in place.
    --ascii             Use ASCII operator aliases (->  !=  <=  etc.).
    --diff              Show unified diff of changes instead of rewriting.

────────────────────────────────────────────────────────────────────────────────
  6.5  nv doc — Generate Documentation
────────────────────────────────────────────────────────────────────────────────

  Usage:  nv doc [options] [<path>]

  Behavior:
    - Scans all .nv source files for /** doc comments */ attached to pub
      declarations.
    - Generates an HTML (or Markdown) documentation site.
    - Supports @param, @returns, @throws, @example tags inside doc comments.
    - Cross-links types and functions automatically.

  Output structure (generated into ./docs/ by default):
    index.html        Module index
    <module>.html     Per-module page with all pub declarations
    search.js         Client-side search index

  Options:
    -o <dir>          Output directory (default: ./docs/).
    --format <fmt>    html (default) | markdown | json (machine-readable).
    --include-private Include non-pub declarations.
    --base-url <url>  For hosted docs (affects links).

────────────────────────────────────────────────────────────────────────────────
  6.6  nv pkg — Package Manager
────────────────────────────────────────────────────────────────────────────────

  Usage:  nv pkg <subcommand> [options]

  Philosophy:
    - Packages are Git repositories (content-addressed by tag/commit SHA).
    - No central registry required; any Git URL works.
    - An optional public registry (pkg.nordvest.dev) provides discoverability.
    - Lock files (nv.lock) ensure reproducible builds.
    - Semantic versioning (semver) for version constraints.

  Package manifest file: nv.pkg (TOML-like key-value format)

    name    = "myapp"
    version = "1.0.0"
    authors = ["Alice <alice@example.com>"]
    license = "MIT"

    [deps]
    "github.com/someone/mylib" = "^1.2.0"
    "github.com/other/utils"   = "0.5.1"

    [dev-deps]
    "github.com/someone/testutils" = "*"

    # Optional feature flags — enable with nv build --feature ssl
    [features]
    ssl = ["github.com/someone/tls@^1.0.0"]
    gpu = []

    # Platform-conditional dependencies
    [target.linux.deps]
    "github.com/someone/epoll-utils" = "^0.3.0"

    [target.darwin.deps]
    "github.com/someone/kqueue-utils" = "^0.3.0"

    [target.windows.deps]
    "github.com/someone/iocp-utils" = "^0.2.0"

  Subcommands:

    nv pkg init                    Create a new nv.pkg in the current directory.
    nv pkg add <url[@version]>     Add a dependency and update nv.lock.
    nv pkg remove <name>           Remove a dependency.
    nv pkg install                 Fetch and build all dependencies.
    nv pkg update [<name>]         Update one or all deps to latest matching version.
    nv pkg list                    List installed packages and their versions.
    nv pkg search <query>          Search the public registry.
    nv pkg publish                 Tag and publish the current package to the registry.
    nv pkg audit                   Check for known security advisories.
    nv pkg vendor                  Copy all resolved deps into vendor/ for offline /
                                   air-gapped builds. vendor/ takes precedence over
                                   the cache when present. Commit vendor/ to pin deps.
    nv pkg sign                    Sign the current package with the author's key via
                                   Sigstore (transparency log). nv pkg install verifies
                                   signatures when the registry provides them.

  Dependency resolution:
    - Uses a SAT-based resolver (PubGrub algorithm) for conflict detection.
    - All transitive deps are recorded in nv.lock with exact SHA references.
    - nv.lock must be committed to version control for application projects.
    - Library projects should not commit nv.lock (so users get fresh resolution).
    - Platform-conditional deps ([target.X.deps]) are resolved only for the
      current platform unless cross-compiling (nv build --target).
    - Feature deps ([features]) are opt-in; resolved only when the feature is
      explicitly enabled via nv build --feature <name>.

  Workspace support:
    - A workspace root nv.pkg with a [workspace] section lists member packages.
    - nv build / nv test at the workspace root builds/tests all members.
    - Shared deps are deduplicated across workspace members.

  6.8  nv link — Link LLVM IR Libraries (Phase 7, planned)
────────────────────────────────────────────────────────────────────────────────

  Usage:  nv link [options] <file.ll>... -o <output>

  Rationale:
    LLVM IR (.ll) is the portable, platform-independent intermediate
    representation produced by every nv build --emit-llvm invocation.
    nv link provides a first-class way to combine multiple .ll modules —
    whether produced by nv, by hand, or by a third-party compiler — into a
    single native binary. This enables:

      - Pre-compiling large, stable sub-modules to .ll and caching them
        independently of the application code (faster incremental builds).
      - Distributing library packages as .ll artifacts alongside .nv sources
        (allows faster install-time builds; .ll is already type-checked).
      - FFI glue: hand-authored LLVM IR optimizations or intrinsics that
        cannot be expressed in Nordvest can be slotted in via nv link.
      - Hermetic builds: vendor/ directories can contain .ll snapshots that
        reproduce exactly without recompiling dependencies from source.

  Behavior:
    - Invokes llvm-link to merge all input .ll files into a single IR module.
    - Passes the merged module to clang to produce the final native binary.
    - The Nordvest runtime (inline IR functions) is expected to already be
      present in at least one of the input files, or linked via --link-rt.
    - Equivalent pipeline:
        llvm-link <files...> -o merged.ll
        clang -x ir merged.ll -o <output> [-O2] [-g] [-static]

  Options:
    -o <output>         Output binary name (required).
    --release           Pass -O2 to clang.
    --debug             Debug build with DWARF info.
    --static            Statically link the runtime.
    --emit-ll           Emit merged .ll only; do not invoke clang.
    --link-rt           Inject the Nordvest runtime IR before merging (for
                        .ll files produced outside of nv build).

  Integration with nv build:
    - nv build --emit-llvm -o foo.ll compiles a single module to .ll.
    - nv build --lib --emit-llvm -o libfoo.ll compiles a library to .ll.
    - nv link libfoo.ll libbar.ll main.ll -o myapp combines them.
    - The --link-ll flag on nv build is syntactic sugar for this pipeline:
        nv build main.nv --link-ll extra.ll -o myapp
      compiles main.nv to IR, merges extra.ll, then links.

================================================================================
  APPENDIX: KEY DESIGN DECISIONS TO REVISIT POST-v1
================================================================================

  A.  Memory management strategy
      Current plan: reference counting (RC) with cycle detection via a
      background thread (like Swift/Python). Revisit: ownership types
      (Rust-style borrow checker) once the language is self-hosted and can
      bear the complexity of the type system extension.

  B.  Compile-time metaprogramming
      Chapter 17's @derive/@builder/@newtype/@lazy/by system is the v1
      answer for the common structural-boilerplate cases. Post-v1, a user-
      extensible macro system (hygienic, type-checked) would allow library
      authors to define their own derive traits and code-gen annotations.
      Defer until self-hosting so the macro expander can be written in
      Nordvest. The std.json @Json annotation is a bridge case: implemented
      as a compiler-known annotation in v1, replaceable by a user macro
      once the macro system lands.

  C.  Ownership and ptr<T> safety
      Currently ptr<T> is "always unsafe". A future region/lifetime system
      could allow safe raw pointers in restricted contexts. Keep the unsafe
      keyword as a marker even in v1 so the audit trail is clear.

  D.  Operator overloading scope
      Currently: +, -, *, /, %, ^, ==, <, >, <=, >= inside class bodies.
      Revisit: custom infix operators (Haskell-style) — defer to post-v1 to
      avoid parsing complexity during the bootstrap phase.

  E.  Package registry governance
      A public registry requires spam prevention, namespace squatting policy,
      and DMCA compliance. Design this carefully in Phase 3 before launch;
      do not rush it.

  F.  WASM target
      A WebAssembly backend (via LLVM's wasm32-unknown-unknown target) would
      enable browser-side Nordvest and the "Try Nordvest" playground. The LLVM
      backend makes this relatively cheap to add in Phase 3.

  G.  Integer-division token ambiguity
      The two-slash // sequence creates a lexer ambiguity with line comments.
      Resolution: ÷ (U+00F7) is the sole integer-division token.
      // is ALWAYS a line comment in Nordvest, with no exceptions.
      There is no // integer-division operator.

  H.  ? disambiguation
      The ? character has exactly two uses: nullable type modifier (T?) and
      Result propagation (expr?). There is no ternary operator — use inline
      if/then/else instead. This eliminates the three-way disambiguation
      complexity entirely. The two remaining uses are structurally distinct:
      T? appears in type position; expr? appears as a postfix on an expression.

  I.  Raw / multi-line strings
      r"..." is a raw string literal: no escape processing, no interpolation.
      Backslashes are literal; braces {} are literal. For multi-line raw strings,
      r"""...""" uses triple double-quotes; the leading newline after the opening
      """ is stripped, and indentation is stripped to the level of the closing
      """. These are Phase 1 features (see A.4).

  J.  Operator overloading scope (extended)
      Bitwise operators &, |, ⊕ are now user-overloadable alongside arithmetic
      operators. The full set inside class bodies: +, -, *, /, ÷, %, ^, &, |,
      ⊕, ==, <, >, ≤, ≥. Revisit: compound operators (&=, |=, ⊕=, <<=, >>=)
      are auto-derived from their base forms (a &= b  →  a = a & b).

────────────────────────────────────────────────────────────────────────────────
  7.1  VS CODE EXTENSION
────────────────────────────────────────────────────────────────────────────────

Repository:  nordvest-lang/nv-vscode
Language:    TypeScript (vscode extension API + vscode-languageclient)
Marketplace: publish to VS Code Marketplace under publisher "nordvest-lang"
Phase:       initial release in Phase 3; polish in Phase 4

  7.1.1  File type & syntax highlighting
         - Register the .nv file extension and "nordvest" language ID.
         - Ship a TextMate grammar (syntaxes/nordvest.tmLanguage.json) for
           offline highlighting (used when nv-lsp is not running, and in
           GitHub/GitLab diffs via the Linguist grammar).
         - TextMate scope map:
             keyword.control.nv       fn if else match for while return guard
             keyword.other.nv         let var import module pub extend sealed by
             keyword.annotation.nv    @derive @builder @newtype @lazy @if @else
                                      @gpu @extern @CLayout @asm @bytes @c @cpp
             storage.type.nv          int float bool str byte char int64 float32
             constant.language.nv     true false nil π e ∞
             constant.numeric.nv      integer and float literals, _ separators
             string.quoted.double.nv  "…" including {interpolation} spans
             variable.language.nv     self
             punctuation.arrow.nv     → ->
             keyword.operator.math.nv ∀ ∃ ∑ ∏ ∈ ∧ ∨ ¬ ≤ ≥ ≠
             comment.line.nv          //
             comment.block.nv         /* */  /** */
         - Bracket pairs: () [] [[]] {} — enables auto-close and rainbow brackets.
         - Indentation rules: increaseIndentPattern after fn/class/if/else/match
           etc. (no trailing colon needed; indentation is structural).

  7.1.2  LSP client integration
         - Use vscode-languageclient to launch nv-lsp as a child process.
         - Locate nv-lsp via: (1) nv-lsp in PATH, (2) extension-bundled binary
           (platform-specific vsix), (3) user-configured path in settings.
         - Restart server command via command palette: "Nordvest: Restart Server".
         - Status bar item shows server state (starting / ready / error) and
           the Nordvest version in use.

  7.1.3  Unicode input helper
         Nordvest's mathematical operators are Unicode characters that cannot be
         typed on a standard keyboard. The extension provides two input methods:

         a) Backslash escape sequences (default, like LaTeX):
              \forall  →  ∀       \exists  →  ∃       \sum     →  ∑
              \prod    →  ∏       \in      →  ∈       \and     →  ∧
              \or      →  ∨       \not     →  ¬       \leq     →  ≤
              \geq     →  ≥       \neq     →  ≠       \pi      →  π
              \inf     →  ∞       \->      →  →       \e       →  e (math e)
            Triggered on space or operator character after a recognized sequence.

         b) Completion items: typing the ASCII alternative (e.g. >=, !=, ->)
            offers a completion that inserts the Unicode form. Acceptance is
            optional — ASCII alternatives remain valid Nordvest.

  7.1.4  Editor integrations
         - Format on save (delegates to nv-lsp / nv fmt).
         - Run current file: keybinding (Ctrl+F5 / Cmd+F5) triggers
           "nv run ${file}" in the integrated terminal.
         - Problem matcher for nv output errors so they appear in the Problems
           panel and inline in the editor gutter.
         - Tasks: auto-generate tasks.json entries for nv build, nv test, nv run.
         - Snippets: fn, class, record, sealed, match, if let, guard let, for,
           while, import, module, Result, async fn — available via snippet prefix.

  7.1.5  Release & packaging
         - Platform-specific vsix bundles include pre-built nv-lsp binaries for
           linux-x64, linux-arm64, darwin-arm64, darwin-x64, win32-x64.
         - A universal vsix (without bundled binary) is published for users who
           install nv separately.
         - Versioning: extension version tracks Nordvest language version
           (e.g. extension 0.3.x supports language spec 0.3).

────────────────────────────────────────────────────────────────────────────────
  7.2  INTELLIJ PLATFORM PLUGIN
────────────────────────────────────────────────────────────────────────────────

Repository:  nordvest-lang/nv-intellij
Language:    Kotlin (IntelliJ Platform SDK + Gradle IntelliJ Plugin)
Marketplace: JetBrains Marketplace under "Nordvest"
Target IDEs: IntelliJ IDEA (Community & Ultimate), CLion, PyCharm, all other
             JetBrains IDEs via the shared IntelliJ Platform
Phase:       initial release Phase 3; PSI-full implementation Phase 4

  Implementation approach — two-tier:

  Tier 1 (Phase 3, ships with M3): LSP4IJ backend
    Use the LSP4IJ open-source IntelliJ LSP client library to wire nv-lsp into
    IntelliJ. This gives diagnostics, completion, go-to-definition, hover,
    formatting, and rename with low implementation cost. Combine with a hand-
    written lexer for syntax highlighting (LSP semantic tokens are available
    but too slow for keystroke-level highlighting in IntelliJ).

  Tier 2 (Phase 4): full PSI plugin
    Replace LSP4IJ with a native IntelliJ PSI (Program Structure Interface)
    implementation. This enables the deeper integrations only PSI can provide
    (structural search & replace, extract refactoring, intention actions,
    cross-project find usages, debugger). Keep nv-lsp as the type-checking
    source of truth; the PSI layer parses and indexes structure without
    repeating type inference.

  7.2.1  File type and language registration
         - Register NordvestFileType (.nv, icon: nordvest-lang logo) and
           NordvestLanguage (ID: "Nordvest").
         - Associate .nv with the Nordvest language in IntelliJ's file type
           settings; users can override if needed.

  7.2.2  Lexer and syntax highlighting
         - Write a hand-rolled IntelliJ Lexer (implementing com.intellij.lexer.Lexer
           directly, not via JFlex, to keep Unicode operator handling explicit).
         - Token types cover the same scopes as the VS Code TextMate grammar.
         - The SyntaxHighlighter maps token types to TextAttributesKey values;
           the default scheme mirrors IntelliJ's built-in Kotlin scheme so that
           Nordvest looks at home in the IDE without extra configuration.
         - Color scheme settings page: "Editor → Colors → Nordvest" lets users
           customize per-token colors.
         - Brace/bracket matching: (), [], [[]], {} pairs including the
           asymmetric range delimiters [ and ] used in range notation.

  7.2.3  PSI tree (Tier 2)
         - Define PsiElementTypes and IElementTypes for every AST node.
         - Implement a PsiParser backed by the same recursive-descent logic as
           the compiler parser (ported to Kotlin PSI stubs). The compiler and
           IDE share the same grammar productions to prevent drift.
         - Stub index: index top-level declarations (fn, class, record, sealed,
           interface, module) for fast workspace-wide symbol lookup without
           parsing every file.

  7.2.4  Completion (Tier 2)
         - CompletionContributor: provides keyword completions, top-level
           symbol completions from the stub index, and member completions from
           the resolved type of the receiver expression.
         - Unicode operator completions: type \forall, get ∀ inserted. Same
           backslash-escape table as the VS Code extension.
         - Smart completion (Ctrl+Shift+Space): filters completions to types
           compatible with the expected type at the caret.

  7.2.5  Inspections and quick-fixes
         - "Redundant nullable" — T? in a context that can never be nil.
         - "Missing match case" — offer to add the missing sealed variants.
         - "Replace -> with →" (and vice-versa with a project-level setting).
         - "Convert ASCII operator to Unicode" — >=, !=, -> → ≥, ≠, →.
         - "Introduce variable" refactoring (extract subexpression to let).
         - "Add ? propagation" — wrap a Result call site with ?.

  7.2.6  Run configurations
         - NordvestRunConfigurationType: "nv run <file>" with working directory,
           environment variables, and program arguments fields.
         - NordvestTestRunConfigurationType: "nv test [filter]" with test name
           filter and output options.
         - Run gutter icons next to fn main() and fn test*() functions.

  7.2.7  nv tool integration
         - Detect nv binary via PATH or a project-level SDK setting
           (Settings → Languages & Frameworks → Nordvest SDK).
         - "Download and install" action in the SDK settings opens the official
           download page or uses a bundled installer script.
         - File watcher: invoke "nv fmt" on save (configurable; off by default).
         - Build tool window: parse nv build output and populate the "Problems"
           tool window with file-linked errors.

  7.2.8  Debugger (Phase 4 stretch)
         - Nordvest emits DWARF debug info via LLVM; CLion and IntelliJ with
           the Native Debugging Support plugin can attach to nv-built binaries.
         - A NordvestDebugProcess adapts IntelliJ's XDebugger to the DWARF
           symbol data, enabling breakpoints, step-through, and variable
           inspection in .nv source files.

────────────────────────────────────────────────────────────────────────────────
  7.3  SHARED ASSETS & GOVERNANCE
────────────────────────────────────────────────────────────────────────────────

  7.3.1  Shared TextMate / tmLanguage grammar
         The same nordvest.tmLanguage.json is used by the VS Code extension,
         the Linguist grammar (GitHub syntax highlighting), and as a reference
         for the IntelliJ syntax highlighter. It lives in a separate repository
         nordvest-lang/nv-textmate-grammar, published to the TextMate / Linguist
         grammar registries, and consumed as a submodule/package by both plugins.

  7.3.2  Icon and branding
         The .nv file icon, language logo, and color palette are defined once in
         nordvest-lang/nv-assets. Both plugins import from there to keep branding
         consistent across editors and the documentation site.

  7.3.3  Versioning and compatibility
         - Plugin versions track language spec versions (semver minor = language
           minor; patch = plugin-only fixes).
         - Each plugin declares a minimum nv-lsp version and a minimum IDE
           version it supports. Older nv-lsp versions degrade gracefully (missing
           extended LSP features are silently skipped).

  7.3.4  Testing strategy
         - VS Code: Jest + @vscode/test-electron for extension tests; a headless
           VS Code instance runs integration tests against a synthetic .nv
           workspace.
         - IntelliJ: the IntelliJ Platform Test Framework (LightPlatformTestCase /
           CodeInsightTestFixture) drives parser, highlighter, completion, and
           inspection tests against known .nv files.
         - Both use the same golden .nv test corpus as the compiler test suite
           (from tests/ in the main repo) to prevent grammar drift.
